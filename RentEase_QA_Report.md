
# RentEase QA Report — July 22, 2026 (v2, live-tested)

## Testing method

v1 of this report (see revision note at bottom) was a static code trace because the sandbox couldn't run the dev server. Every Critical/High item below was reproduced live, not just read in code — where I also traced the root cause in source, I show both.

## What's still solid

Tenant registration works end-to-end live: filled the form, submitted, got redirected and properly logged in with session cookie set correctly, nav updated to show "Dashboard"/"Log Out". The one real Firestore-backed listing ("Sunlit 2-Bed Apartment near Downtown") loads correctly from the `/properties` grid through to its detail page with no errors. Login page and the sign-in/create-account tab UI render cleanly.

## Caveat

I could not get the browser extension to actually resize the rendered viewport in this environment (the resize call reported success but `window.innerWidth` stayed at 1920px), so true mobile/tablet responsive testing wasn't completed live this pass — flagging honestly rather than guessing. The Stripe webhook/payment items also couldn't be safely re-tested live without real Stripe test credentials.

## Summary

Both bugs you reported are now root-caused: the "page not found" click is every homepage showcase card linking to a fake mock property ID that 404s against the real database, and the "refresh goes to footer" issue is a scroll-restoration effect that overshoots the page height and gets clamped to the very bottom. On top of that, live testing surfaced a serious access-control gap (two owner pages with zero auth check) and, notably, a real registered owner account in your live database that's currently locked out of its own dashboard, listings, and "add listing" pages because of a corrupted role value.

**Top 5 priority fixes:**
1. Fix the homepage showcase cards — either point "View Full Listing" at real listings or remove the dead-end mock cards (#1).
2. Fix the intro-scroll `scrollTo` target so it can't exceed document height (clamp it to the actual hero-section height, not a fixed viewport multiplier) (#2).
3. Clean up the corrupted `"owner\n"` role value for the affected account, and make role checks resilient (`.trim()`/case-insensitive) so a single bad character can't lock an owner out of their own portal (#3).
4. Add the missing auth/role guard to `/owner/analytics` and `/owner/settings` (#4).
5. Verify the Stripe webhook signature before trusting any event, and decide whether the booking system ships this version (#7, #8, #9).

**Launch-readiness verdict: Not ready.** Two of the platform's most basic interactions — clicking a homepage listing and refreshing the page — are broken for every visitor, and there's at least one real account currently locked out of core owner functionality due to a data bug. These need to be fixed before this goes in front of real users.
